Every request to the Pizza API must include a valid API key in the `Authorization` header. There are no session cookies, OAuth flows, or username/password combinations — just a single key passed as a Bearer token. This keeps integration straightforward while still giving you full control over access and rotation.

## Getting your API key

<Steps>
  <Step title="Sign up or log in">
    Go to [https://dashboard.pizzaapi.com](https://dashboard.pizzaapi.com) and create an account or sign in to your existing one.
  </Step>
  <Step title="Navigate to API Keys">
    In the left sidebar, open **Settings** then click **API Keys** to reach the key management page.
  </Step>
  <Step title="Generate and copy your key">
    Click **Generate New Key**, give it a descriptive name (e.g. `production-web-app`), then copy the key immediately. For security reasons, the full key value is **only displayed once** at the moment of creation.
  </Step>
</Steps>

<Warning>
  Treat your API key like a password. Never share it publicly, commit it to version control, or expose it in client-side code. If a key is compromised, revoke it immediately from the dashboard and generate a replacement.
</Warning>

## Making authenticated requests

Pass your API key as a **Bearer token** in the `Authorization` header on every request. No other authentication mechanism is accepted.

<CodeGroup>

```bash curl
curl https://api.pizzaapi.com/v1/menu \
  -H "Authorization: Bearer YOUR_API_KEY"
```

```javascript JavaScript
const response = await fetch("https://api.pizzaapi.com/v1/menu", {
  headers: {
    Authorization: "Bearer YOUR_API_KEY",
  },
});

const data = await response.json();
```

```python Python
import requests

response = requests.get(
    "https://api.pizzaapi.com/v1/menu",
    headers={"Authorization": "Bearer YOUR_API_KEY"},
)

data = response.json()
```

</CodeGroup>

## Test vs. live keys

Pizza API provides two classes of API keys so you can develop and test safely without affecting real operations.

| Key prefix | Environment | Behaviour |
|---|---|---|
| `pk_test_` | Test | Simulates the full API flow; no real deliveries are dispatched. |
| `pk_live_` | Live | Triggers real orders and real deliveries. Charges apply. |

<Note>
  Always develop and debug with a `pk_test_` key. Switch to your `pk_live_` key only when you're ready to accept real orders. Test and live keys are managed separately in the dashboard.
</Note>

## Authentication errors

When authentication fails, the API returns a standard JSON error body alongside an HTTP error status.

| Status | Meaning |
|---|---|
| `401 Unauthorized` | The `Authorization` header is missing, malformed, or the key is invalid. |
| `403 Forbidden` | The key is valid but does not have permission to access the requested resource. |

Both errors follow the same error envelope format:

```json Example 401 error response
{
  "error": {
    "code": "unauthorized",
    "message": "Invalid API key. Check your key and try again."
  }
}
```

If you receive a `403`, verify that your key has the required scopes enabled. You can review and update key permissions from the **API Keys** page in the dashboard.

## Security best practices

Following these practices will keep your integration secure and your customers' data protected.

- **Use environment variables.** Store your API key in an environment variable (e.g. `PIZZA_API_KEY`) and read it at runtime. Never hardcode it directly in source files.
- **Never expose keys client-side.** API calls that include your secret key must originate from your server. Bundling a key into browser JavaScript or a mobile app binary makes it trivially extractable.
- **Rotate compromised keys immediately.** If you suspect a key has been exposed, go to **Settings → API Keys** in the dashboard, revoke the affected key, and generate a new one. Update your environment variables before redeploying.
- **Use separate keys per environment.** Maintain distinct keys for development, staging, and production so that a leaked development key never puts your live data at risk.

<Tip>
  Most deployment platforms — such as Vercel, Railway, Render, and AWS — provide a first-class secrets manager or environment variable store. Set `PIZZA_API_KEY` there rather than in a `.env` file that could accidentally be committed to your repository.
</Tip>
